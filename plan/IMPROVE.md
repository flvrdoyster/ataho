## 1. 드로우 로직 보정 (캐릭터별 운/개성 반영)
- **현재 상태:** `generateDeck`에서 전체 패를 섞고 순서대로 뽑는 완전 무작위 방식.
- **개선 방향 (후보군 평가 방식):** 
  - 캐릭터의 `luck` 스탯에 따라 덱의 맨 위에서 N장의 패를 몰래 살펴보고, 현재 손패(Hand)와 비교해 가장 가치 있는(유효패) 패를 골라 드로우하는 시스템 도입.
  - 남은 패는 다시 덱의 원래 위치에 돌려놓거나 섞어서 전체 패의 구성비를 훼손하지 않음.
- **기대 효과:** 캐릭터 설명에 맞는 '패를 뽑는 운'이 시각적으로/체감상으로 구현됨.

## 2. 상황 인지형 캐릭터 대사 (최적화)
- **현재 상태:** 패를 버릴 때 출력되는 대사가 상황(필요한 패를 버리는지, 쓸모없는 패를 버리는지)과 무관하게 랜덤 출력됨. 과거 상황 인지 로직을 넣었으나 성능 저하(프레임 드랍 등)로 기각.
- **개선 방향:**
  - 무거운 텐파이/샹텐(Shanten) 계산 로직을 매번 돌리지 않고, **가벼운 휴리스틱(Heuristics)** 기반으로 패의 가치를 평가하여 대사를 분기.
    - 예: "내가 버린 패의 숫자 ±2 범위 내에 내 손패가 존재하는가?" 단 1번의 가벼운 루프 사용 (O(N) 연산).
  - 필요시 평가 결과를 캐싱(Memoization)하거나 드로우/타패 시점에 딱 1번만 계산하여 메인 스레드 부하 최소화.
- **기대 효과:** 버벅임 없이 자연스럽게 캐릭터가 현재 상황(패가 말림, 좋은 패가 들어옴 등)에 반응하는 생동감 있는 대사 출력.

## 3. 오디오 시스템 개선 (모바일 핵심 병목) ← 현재 작업
- **현재 상태 (문제 2가지):**
  1. **SFX 씹힘:** `HTMLAudioElement` + `cloneNode()` 풀 방식. 모바일에서 `play()` 호출 시마다 디코딩 파이프라인이 메인 스레드에 묻히고, `currentTime = 0` 리셋이 재디코딩을 트리거함.
  2. **BGM 미재생:** `audio.play()` 실패 시 Promise `.catch()` 안에서 재시도하는데, iOS Safari는 유저 제스처와 동기적으로 연결되지 않은 async 콜백에서의 `play()`를 거부함. 또한 씬 전환 시 `isWaitingForInteraction` 플래그 경쟁으로 리스너가 등록되지 않는 케이스 존재.
- **개선 방향 (`assets.js` 단일 파일 변경):**
  - **SFX → Web Audio API 전환:** 로드 시 `fetch + AudioContext.decodeAudioData()`로 한 번만 디코딩 → `AudioBuffer` 저장. 재생 시 `createBufferSource().start()` — 디코딩 없이 즉시 재생, 풀링 불필요.
  - **BGM 미재생 → `AudioContext` 잠금 해제 패턴:** 첫 번째 유저 제스처(`touchstart`/`click`) 시점에 `audioContext.resume()`을 동기적으로 한 번 호출. 이후 `HTMLAudioElement.play()`는 context가 `running` 상태이므로 async 여부 무관하게 동작. BGM은 긴 파일이라 스트리밍이 유리하므로 `HTMLAudioElement` 유지.
- **변경 범위:** `assets.js` 하나.

## 5. 전반적인 게임 퍼포먼스 최적화 (지속 과제)

### 프로파일링 및 코드 분석 결과 (2026-04-27)

코드를 직접 읽은 결과, 원래 우려했던 일부 문제는 이미 잘 구현되어 있고, 실제 점검이 필요한 지점이 좁혀졌다.

**이미 잘 되어 있는 것 (건드리지 않음):**
- 게임 루프: `requestAnimationFrame` 기반, delta-time 캡 처리 완비.
- 타이머: 엔진 자체 tick 기반 `setTimeout` 사용 (네이티브 `setTimeout` 없음).
- 오디오: SFX 풀링(cloneNode, 최대 5개), BGM 단일 인스턴스 — 구조 양호.
- 렌더링: 순수 Canvas 2D API (DOM 리플로우 없음), bgCanvas/fgCanvas 레이어 분리로 정적 UI는 dirty 시에만 재렌더.
- `measureText`: `drawCharacterNames` (static 재렌더 시만 호출) + 스킬 확인창(STATE_TILE_EXCHANGE)에서만 사용 — 매 프레임 호출 아님.

**실제 점검 대상 (우선순위 순):**

1. **`battleRenderer.js` draw() 내 per-frame 연산 비용**
   - `drawDiscards`, `drawOpenSets`, `drawBuffIndicators`, `drawDialogue` 등이 매 프레임 호출됨.
   - 내부에서 `ctx.save()`/`restore()` 26쌍, 배열 탐색 19건 존재 — 실제 프레임 비용은 프로파일링으로 측정 필요.
   - 특히 `getVisualMetrics`가 매 프레임 계산되는지 확인 필요.

2. **`battleEngine.js` update() 내 상태별 로직 비용**
   - 3,236줄 짜리 상태 머신 — 각 state의 update 경로가 최소한의 연산만 수행하는지 확인.
   - `updateTimeouts`는 O(N) 역순 루프로 구현, 구조 양호. 단, timeout 수가 급증하는 상황이 있는지 확인.

3. **GC 압박 (단명 객체 생성)**
   - 핫 패스(매 프레임 draw/update)에서 임시 객체/배열이 생성되면 GC가 일정 주기로 프레임을 끊음.
   - Chrome DevTools의 Memory 탭 Heap Snapshot 비교 또는 Performance 탭의 GC 마커로 확인.

**프로파일링 결과 (2026-04-27, 4x CPU 쓰로틀링):**
- `loop()` 평균 **0.47ms**, 16ms 초과 1회/2544회 — JS 로직 자체에 병목 없음.
- 307ms Long Task는 `CpuProfiler::StartProfiling` 아티팩트 (DevTools 측정 노이즈) — 무시.
- GC: 쓰로틀링 없이 74회 → **쓰로틀링 시 107회, 총 ~60ms** (MajorGC 5회 포함). 핫 패스에서 임시 객체가 지속 생성되고 있음을 시사.

**재조정된 최적화 방향:**
- JS 로직 전면 최적화는 효과 대비 비용이 크므로 보류.
- **GC 압박 감소**를 단일 타겟으로 설정: `battleRenderer.draw()` 및 `battleEngine.update()` 핫 패스에서 매 프레임 생성되는 임시 배열/객체를 특정하여 제거.
- 체감 버벅임의 원인이 JS 외부(Canvas GPU 합성, 오디오 스레드 등)일 수 있으므로, **#3 오디오 개선 완료 후 모바일에서 재평가**하여 잔존 증상 여부를 판단하고 다음 타겟 결정.

## 4. 모바일 환경 터치 조작감 개선 (Mobile UX/UI)
- **현재 상태:** PC 마우스 조작을 우선으로 설계되어, 모바일 브라우저에서 터치 시 조작이 애매하거나 오터치가 발생하기 쉬움 (예: 패 선택 시의 딜레이, 좁은 터치 영역 등).
- **개선 방향:**
  - **터치 이벤트 최적화:** `click` 이벤트 외에 `touchstart`, `touchend` 이벤트를 적절히 병행 처리하여 모바일 환경 특유의 클릭 지연(300ms 딜레이)을 최소화.
  - **터치 영역(Hitbox) 확장:** 패(Tile)나 버튼의 보이지 않는 클릭 판정 영역을 시각적 크기보다 조금 더 크게 잡아 오터치를 방지.
  - **상태 피드백 강화:** 터치하는 순간 즉각적인 시각적 피드백(크기 확대, 색상 변화, 햅틱 진동 API 등)을 제공하여 조작의 확신을 줌.
  - **UI 레이아웃 조정 (선택 사항):** 모바일 화면 비율에 맞춰 손가락이 닿기 편한 위치로 메뉴나 버튼 레이아웃을 반응형으로 미세 조정.
